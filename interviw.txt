Interview-ready summary:
In this file, I used Playwright for API testing, request mocking, response interception, UI interaction, assertions, and reusable test setup with beforeEach.

Common interview questions and answers

1. Why did you use beforeEach in this file?
I used beforeEach to handle repeated login and setup logic once for every test. This keeps the code cleaner and avoids duplication.

2. Why use API login instead of UI login?
API login is faster, more stable, and avoids unnecessary UI steps. It helps the test focus on the main scenario.

3. What does page.addInitScript() do here?
It runs code before the page loads. In this file, I used it to store the token in localStorage so the browser starts in an authenticated state.

4. What is the difference between page.route() and page.waitForResponse()?
page.route() is used to intercept and modify or mock requests before the page receives them.
page.waitForResponse() is used to capture and inspect a real response after it happens.

5. Why did you use route.fetch()?
I used route.fetch() to get the original API response first, then modify part of it before sending it back to the browser.

6. Why use route.fulfill()?
route.fulfill() lets me send mocked or modified response data to the application.

7. Why did you use waitForLoadState('networkidle')?
I used it to wait until network calls settle before validating the UI. This helps reduce timing issues.

8. Why use getByRole()?
getByRole() is more stable and better aligned with accessibility than weak CSS selectors.

9. Why use waitForResponse() in article creation?
I used it to capture the backend response and extract the article slug, which I then used for cleanup.

10. Why is cleanup important in API tests?
Cleanup prevents leftover test data from polluting the environment and affecting later tests.

11. What is the difference between mocking API and testing real API?
Mocking checks frontend behavior with controlled data.
Real API testing checks the actual backend integration and contract.

12. What risks or weaknesses do you see in this file?
- Global shared accessToken can be risky in parallel execution
- Hardcoded credentials are not ideal
- External environment dependency can make tests unstable
- networkidle is useful but can sometimes still be flaky

Strong final answer for interview:
This file demonstrates that I can combine UI automation, API requests, request mocking, response interception, authentication setup, and test data cleanup in Playwright.

//DragandDrop

This test automates a drag-and-drop action inside an iframe using Playwright. Since elements inside an iframe are isolated from the main page DOM, I used frameLocator() to switch context into the frame first, then located the draggable and droppable elements and performed the drag-and-drop action. This shows I understand handling embedded documents, which is a common UI automation challenge.”

Interview questions and answers:

What is an iframe in automation?
iframe is an embedded HTML document inside another page. You cannot reliably interact with its elements from the main page context until you switch into that frame.

Why did you use frameLocator()?
I used frameLocator() because Playwright provides it as the cleanest way to target elements inside an iframe. It automatically scopes all locators to that frame.

What problem happens if you do not switch to the iframe?
Playwright will not find the target elements because they are not in the main document DOM. The test usually fails with element not found or timeout errors.

Why is iframe handling important in interviews?
Many real applications use iframes for ads, payment forms, dashboards, or embedded widgets. Interviewers ask this to check whether you understand DOM context switching.

How would you explain this test in one line?
“It validates drag-and-drop behavior inside an iframe by switching into the frame and interacting with elements there.”

Short answer if they ask “what did you learn from this test?”:
“I learned that iframe elements require separate context handling, and Playwright’s frameLocator() makes that reliable and readable.”

Here are 10 iframe-focused interview questions with concise spoken answers based on drag&Drop.spec.js.

What does this test do?
“This test verifies drag-and-drop functionality inside an iframe. I first switch to the iframe context using Playwright, then interact with the source and target elements inside that frame.”

Why is iframe handling different from normal element interaction?
“Because iframe content is isolated from the main page DOM. Elements inside it are not directly accessible from the parent page, so automation tools need explicit frame context.”

Why did you use frameLocator() instead of a normal locator?
“frameLocator() is the recommended Playwright approach for iframe elements. It keeps the code cleaner and automatically scopes all child locators to that specific frame.”

What happens if you try to locate iframe elements without switching context?
“The test will usually fail with timeout or element-not-found errors, because Playwright is searching the main page DOM, not the iframe DOM.”

How do you identify the correct iframe?
“I inspect the page and choose a stable selector for the iframe, such as an attribute, title, id, or another reliable parent-child relationship. In this test, I used the iframe under the Photo Manager section.”

How would you make this iframe test more stable?

How do you explain drag-and-drop inside an iframe in an interview?
“I explain that there are two challenges: first switching into the iframe context, and then performing the UI interaction. The key idea is that the drag-and-drop logic is normal once the frame context is correct.”

Here are tougher iframe follow-up questions with interview-ready answers.

What is the difference between frameLocator() and page.frame() in Playwright?
frameLocator() is locator-based and generally cleaner for writing stable tests. page.frame() gives direct access to a frame object, which can be useful when selecting a frame by name or URL, but it is usually less ergonomic than frameLocator() for element interactions.

When would you choose page.frame() over frameLocator()?
I would use page.frame() when I need to inspect or identify a frame dynamically by URL, name, or other frame metadata. For normal UI interaction inside a known iframe, I prefer frameLocator() because it keeps the test more readable.

How do you handle dynamically loaded iframes?
I wait for the iframe element to appear first, then use frameLocator() and wait for a stable element inside the frame before interacting. I avoid fixed delays and rely on Playwright’s waiting behavior plus assertions.

How would you test nested iframes?
I would chain the frame access step by step. First access the parent iframe, then target the child iframe inside it, then locate the element inside the nested frame. The key is respecting each DOM boundary.

What if the iframe selector is unstable?
I would try to identify a better stable attribute such as title, name, id, or a nearby section with predictable structure. If none exists, I would work with developers to add test-friendly selectors.

Can Playwright interact with cross-origin iframes?
Yes, but with limitations depending on the browser and the type of interaction. Playwright can still automate many actions through frame APIs, but browser security rules can affect script access patterns, so I rely on Playwright-supported interactions rather than assuming full DOM freedom.

Why do iframe tests fail more often than normal tests?
Because they add another level of context and timing. Failures often come from the frame not being ready, weak selectors, dynamic loading, or trying to access frame content from the wrong context.

How would you debug a failing iframe test?
I would inspect whether the iframe is present, confirm the selector targets the right frame, check whether the frame content has loaded, and use Playwright trace, screenshots, or headed mode to see where the failure happens.

How do you verify drag-and-drop actually succeeded inside an iframe?
I would add a post-action assertion, such as checking the dropped element’s new location, a success message, changed DOM state, or class update. I do not rely only on the drag action itself.

What is the biggest mistake candidates make with iframe automation?
They treat iframe elements like normal page elements and forget the context switch. The second common mistake is using brittle selectors and fixed waits instead of proper synchronization.

How would you design a reusable helper for iframe interactions?
I would create a helper that accepts the frame selector and returns a scoped locator or helper object for actions inside that frame. That reduces duplication and makes tests more maintainable when multiple specs use the same iframe.

If an iframe reloads during a test, what would you do?
I would re-establish the frame interaction after the reload and avoid holding stale assumptions about the previous frame state. Then I would wait for a known element inside the refreshed frame before continuing.

How do iframes affect test architecture in larger frameworks?
They usually push you toward better page-object or component-object design. Iframe areas should be modeled as separate components so locators and behaviors stay isolated and reusable.

What would you say if the interviewer asks whether iframe testing is a frontend or backend concern?
It is mainly a frontend automation concern because it affects DOM interaction, selectors, timing, and browser behavior. But it can also reflect system integration issues when the iframe hosts third-party or separately deployed content.

What is the best one-line senior-level answer about iframe testing?
“Iframe automation is mainly about correct context isolation, stable frame targeting, and synchronization after the embedded content is ready.”
